## aden-hive/hive (⭐9,993)

### Links
- **GitHub**: https://github.com/aden-hive/hive
- **Docs**: https://docs.adenhq.com/
- **PyPI**: N/A (uses `uv` workspace layout)

### What It Does (Plain English)
Hive is a **production runtime harness for AI agents**. Instead of just running prompts through an LLM, it wraps agents in a full production environment with state management, crash recovery, cost enforcement, and real-time observability. Users describe goals in natural language; a "coding agent" generates the agent graph and connection code to achieve it. The framework handles failure capture, evolves the graph through the coding agent, and redeploys automatically.

### Architecture Overview

```
User (Goal in Natural Language)
         ↓
┌─────────────────────────────────────────────┐
│         Coding Agent (LLM-driven)           │
│  - Generates agent graph & connection code  │
└─────────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────────┐
│   Hive Harness (production runtime layer)   │
│  - State isolation & checkpoint-based      │
│    crash recovery                           │
│  - Cost enforcement & budget controls      │
│  - Real-time WebSocket observability        │
│  - Human-in-the-loop interruption nodes     │
│  - Dynamic Node Connections (no hardcoded  │
│    edges; connections generated by LLM)     │
└─────────────────────────────────────────────┘
         ↓
┌─────────────────────────────────────────────┐
│        Worker Bees (SDL-wrapped nodes)      │
│  - Shared memory, local RLM memory          │
│  - Monitoring, tools, LLM access            │
└─────────────────────────────────────────────┘
```

**Key directories:**
- `core/framework/` — core agent runtime and graph executor
- `core/frontend/` — web-based UI/dashboard
- `tools/src/aden_tools/` — MCP tools for agent capabilities
- `core/examples/` — example agents

### Key Technical Decisions

1. **SDL-Wrapped Nodes**: Every node gets shared memory, local RLM memory, monitoring, tools, and LLM access out of the box. Nodes are described in natural language and auto-generated.
2. **Dynamic Graph Execution**: No predetermined edges; connection code is generated by any capable LLM based on your goals.
3. **Checkpoint-based Crash Recovery**: Agents can recover from failures by replaying from checkpoints.
4. **Adaptive Evolution**: On failure, the system evolves the graph and redeploys automatically (outcomes evaluation + adaptive agents).
5. **WebSocket Streaming**: Live monitoring of agent execution, decisions, and node-to-node communication.
6. **Multi-Agent Parallel Execution**: Generated graph can execute in parallel across multiple agents.
7. **LLM Flexibility**: Supports 100+ LLM providers (OpenAI, Anthropic, Google Gemini, DeepSeek, Mistral, Groq, OpenRouter, Hive LLM) via LiteLLM.
8. **MCP Tool System**: Built-in MCP tools for agent capabilities, extensible via `tools/src/aden_tools/`.

### Code Snippets

**Agent creation pattern (natural language goal):**
```python
# User defines goal → Coding Agent generates graph + connection code
# Workers execute with full observability
```

**pyproject.toml structure:**
```toml
[tool.uv.workspace]
members = ["core", "tools"]
```
Uses `uv` workspace for monorepo-style dependency management.

### What CrawlForge MUST Steal from This

| Feature | Why It Matters for CrawlForge |
|---------|------------------------------|
| **Dynamic node connections via LLM** | CrawlForge needs to dynamically route between crawl nodes, hero nodes, LLM summarizers, and storage nodes without hardcoding edges |
| **Checkpoint-based crash recovery** | Web crawling is unreliable; agents must resume from checkpoints on failure |
| **Adaptive graph evolution** | When a crawl pattern fails, the system should auto-evolve the extraction graph |
| **Real-time WebSocket observability** | CrawlForge needs live visibility into crawl progress, token usage, and node health |
| **Multi-agent parallel execution** | Multiple pages/URLs can be crawled in parallel |
| **Human-in-the-loop nodes** | When extraction is ambiguous, pause for human confirmation |

### Integration Approach
- Study `core/framework/` for graph execution patterns
- Mirror the checkpoint/recovery model for crawl state persistence
- Adopt MCP tool pattern for extensibility
- Consider WebSocket-based progress streaming for crawl jobs

## vortezwohl/Autono (⭐209)

### Links
- **GitHub**: https://github.com/vortezwohl/Autono
- **Paper**: https://arxiv.org/abs/2504.04650
- **PyPI**: `pip install -U autono`

### What It Does (Plain English)
Autono is a **ReAct-based highly robust autonomous agent framework** designed to solve complex tasks through adaptive decision-making and multi-agent collaboration. Unlike traditional frameworks that rely on fixed workflows generated by LM-based planners, Autono **dynamically generates next actions** during agent execution based on prior trajectories, enhancing robustness. It uses a **temporal abstraction strategy** with a probabilistic penalty mechanism for task abandonment, and provides a **memory transfer mechanism** for shared, dynamically updated memory among agents.

### Architecture Overview

```
ReAct Loop:
  Thought → Action → Observation → Thought → ...
         ↓
┌─────────────────────────────────────┐
│         Agent Core                   │
│  - ReAct reasoning loop             │
│  - Ability system (decorators)      │
│  - Personality enum (PRUDENT/       │
│    INQUISITIVE)                     │
│  - Brain (model + ability access)  │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│  Hooks: BeforeActionTaken /         │
│  AfterActionTaken                   │
│  (observability & intervention)     │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│  MCP Agent Integration              │
│  (Model Context Protocol support)   │
└─────────────────────────────────────┘
```

**Key modules:**
- `autono/ability/` — ability decorators and registry
- `autono/brain/` — brain module (model + ability orchestration)
- `autono/enum/` — enums including `Personality`
- `autono/exception/` — custom exceptions
- `autono/message/` — message types
- `autono/prompt/` — prompt templates

### Key Technical Decisions

1. **ReAct Loop**: Explicit reasoning-action-observation cycle with dynamically generated next actions.
2. **Ability Decorator System** (`@ability`): Functions decorated as abilities can be granted to agents, enabling extensible tool use.
3. **Personality Enum**: `Personality.PRUDENT` (cautious) vs `Personality.INQUISITIVE` (proactive exploration) affect agent behavior.
4. **BeforeActionTaken / AfterActionTaken Hooks**: Allow observation and intervention in decision-making and execution results.
5. **MCP Integration**: `McpAgent` class for accessing MCP tools via stdio or HTTP.
6. **Memory Transfer for Multi-Agent**: Shared memory mechanism among agents for collaborative tasks.
7. **Temporal Abstraction**: Probabilistic penalty for task abandonment, dynamically adjusting probability of task abandonment via penalizing tactics.
8. **Dependencies**: `langchain >= 0.3.14`, `langchain-core >= 0.3.29`, `langchain-community >= 0.3.14`, `langchain-openai >= 0.2.3`, `openai >= 1.59.3`, `datascope >= 1.20.14`, `websockets >= 15.0.1`, `mcp >= 1.6.0`

### Code Snippets

**Ability definition:**
```python
@ability
def calculator(expr: str) -> float:
    # this function only accepts a single math expression
    return simplify(expr)

@ability
def write_file(filename: str, content: str) -> str:
    with open(filename, 'w', encoding='utf-8') as f:
        f.write(content)
    return f'{content} written to {filename}.'
```

**Agent instantiation:**
```python
model = get_openai_model()
agent = Agent(
    abilities=[calculator, write_file],
    brain=model,
    name='Autono',
    personality=Personality.INQUISITIVE
)
agent.assign("Here is a sphere with radius of 9.5 cm...")
response = agent.just_do_it()
```

**MCP integration:**
```python
@mcp_session(mcp_config)
async def run(session, request: str) -> str:
    mcp_agent = await McpAgent(session=session, brain=get_openai_model()).fetch_abilities()
    result = await mcp_agent.assign(request).just_do_it()
    return result.conclusion
```

### What CrawlForge MUST Steal from This

| Feature | Why It Matters for CrawlForge |
|---------|------------------------------|
| **ReAct loop with dynamic action generation** | CrawlForge agents can dynamically decide next crawl/scrape actions based on prior observations |
| **Ability decorator system** | Clean way to define and grant tools (crawl, parse, store) to agents |
| **BeforeAction/AfterAction hooks** | Intercept and log or modify crawl decisions mid-execution |
| **Personality enum** | Agents can be cautious (throttle requests) vs exploratory (discover new patterns) |
| **MCP integration** | Standard protocol for tool extensibility |

### Integration Approach
- Use `@ability` decorator pattern for crawl/parse/storage operations
- Implement `BeforeActionTaken` hook for request throttling and logging
- Implement `AfterActionTaken` hook for result validation and retry logic
- Consider MCP for connecting to external crawl tools

## mpcabete/bombcrypto-bot (⭐811)

### Links
- **GitHub**: https://github.com/mpcabete/bombcrypto-bot
- **README.en.md**: English version available

### What It Does (Plain English)
This is a **game bot for BombCrypto** that uses **template matching** (image comparison) to detect game elements on screen and automate clicking. It saves screenshots of heroes to `/targets/heroes-to-send-home/` and automatically sends exhausted heroes home. The bot uses `opencv` (cv2) and `numpy` for image processing and `pyautogui` for mouse control.

### Architecture Overview

```
Game Screen (Browser Window)
         ↓
┌─────────────────────────────────────┐
│   Image Capture (screenshot)        │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│   OpenCV Template Matching         │
│   cv2.matchTemplate()               │
│   cv2.minMaxLoc() → click coords   │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│   PyAutoGUI Click Automation        │
│   - Click detected elements         │
│   - Configurable thresholds         │
└─────────────────────────────────────┘
         ↓
┌─────────────────────────────────────┐
│   Config via config.yaml            │
│   - Threshold tuning                │
│   - Hero detection sensitivity       │
└─────────────────────────────────────┘
```

**Key files:**
- `index.py` — main bot loop (16,981 bytes)
- `config.yaml` — all tuning parameters
- `src/` — source modules
- `targets/` — template images for matching
- `requirements.txt` — `opencv-python`, `numpy`, `pyautogui`, `PyYAML`

### Key Technical Decisions

1. **Template Matching over ML**: Uses classic computer vision (OpenCV `matchTemplate`) instead of ML-based detection. This is deterministic and explainable.
2. **Threshold-based Decision Making**: Configurable `threshold` (0.0–1.0) controls match sensitivity. Low threshold = more false positives; high threshold = more false negatives.
3. **Configurable per-element Parameters**: Each target image can have its own threshold and match region.
4. **Send Home Feature**: Screenshots of heroes to send home are stored in `targets/heroes-to-send-home/`. Bot loads these at startup and clicks matching heroes, then checks if home button is dark and work button is not dark before clicking.
5. **Troubleshooting Methodology**: Extensive documentation on false positives/negatives and how to tune.

### Code Snippets

**Template matching logic (conceptual):**
```python
import cv2
import numpy as np
import pyautogui

# Load template
hero_template = cv2.imread('targets/hero.png', 0)

# Capture screen
screenshot = pyautogui.screenshot()
screen_gray = cv2.cvtColor(np.array(screenshot), cv2.COLOR_RGB2GRAY)

# Match template
result = cv2.matchTemplate(screen_gray, hero_template, cv2.TM_CCOEFF_NORMED)
_, max_val, _, max_loc = cv2.minMaxLoc(result)

if max_val >= threshold:
    # Click at match location
    pyautogui.click(max_loc)
```

**config.yaml structure:**
```yaml
home:
  home_button_trashold: 0.8
  # threshold for distinguishing dark home button
threshold: 0.9
# general match threshold
```

### What CrawlForge MUST Steal from This

| Feature | Why It Matters for CrawlForge |
|---------|------------------------------|
| **Template matching for UI detection** | CrawlForge can use template matching for detecting UI elements (buttons, forms) before interaction |
| **Threshold tuning methodology** | Systematic approach to balancing false positives vs negatives |
| **Configurable per-element sensitivity** | Different crawl targets may need different matching thresholds |
| **Deterministic detection** | No ML inference needed for simple visual elements |

### Integration Approach
- Use OpenCV template matching for UI element detection in CrawlForge's browser automation
- Implement threshold as a per-target configuration parameter
- Store reference template images for common UI patterns (login buttons, forms, pagination)

## openatx/atx (⭐1,601)

### Links
- **GitHub**: https://github.com/openatx/atx
- **Status**: Repository appears to be deleted/moved (404 on API)

### What It Does (Plain English)
**ATX (Android Testing eXperience)** was a **mobile automation framework** for Android. It supported starting apps, clicking, inputting text, taking screenshots, and performing assertions — all over a JSON HTTP API or Python library. It was the predecessor to `uiautomator2` and inspired many modern mobile automation tools.

### Architecture Overview

```
Computer (Python Client / HTTP API)
         ↓
┌─────────────────────────────────────┐
│   ATX Server (on Android device)    │
│   - uiautomator-based               │
│   - JSON HTTP API                   │
│   - Screen recording                │
│   - Element inspection              │
└─────────────────────────────────────┘
```

**Key features:**
- JSON HTTP API for cross-language support
- Python client library
- App installation and control
- Screenshot and element inspection
- Text input and gestures

### Key Technical Decisions

1. **JSON HTTP API**: Allowed any language to interact with Android devices over HTTP, not just Python.
2. **uiautomator Integration**: Leveraged Android's uiautomator for element inspection and interaction.
3. **Minimal Device Requirements**: Only required the ATX server app on the Android device; computer side was language-agnostic.

### What CrawlForge MUST Steal from This

| Feature | Why It Matters for CrawlForge |
|---------|------------------------------|
| **HTTP API for device control** | CrawlForge could expose a REST API for cross-platform/cross-language automation |
| **Device-agnostic client architecture** | Separate protocol layer from implementation |
| **Minimal device footprint** | Only require a lightweight agent on target devices |

### Integration Approach
- Study the HTTP API design for CrawlForge's crawl protocol
- Consider a lightweight agent + HTTP server architecture for distributed crawling
